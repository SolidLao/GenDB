JOIN ORDER OPTIMIZATION REPORT - Iteration 1
============================================

TASK: Optimize join order for Q3 (customer ⋈ orders ⋈ lineitem)

ANALYSIS:
---------

Current Join Order: customer → orders → lineitem

Cardinality Estimates (SF10):
- customer: 1.5M rows → 300K after filter (c_mktsegment = 'BUILDING', ~20% selectivity)
- orders: 15M rows → 1M after filter (o_orderdate < '1995-03-15', ~33% selectivity)
- lineitem: 60M rows → 40M after filter (l_shipdate > '1995-03-15', ~67% selectivity)

Join Intermediate Sizes:
1. Filter customer: 1.5M → 300K (build filtered_customers hash set)
2. Join customer-orders + filter: 15M × (300K/1.5M) × 0.33 = ~1M → filtered_orders
3. Join orders-lineitem: 60M × 0.67 × (1M/15M) = ~2.7M → final aggregation

Build/Probe Decisions:
- customer-orders join: customer builds (300K), orders probes (15M) ✓ CORRECT
- orders-lineitem join: orders builds (1M), lineitem probes (60M) ✓ CORRECT

Alternative Join Orders Considered:
------------------------------------

Option 1: orders → customer → lineitem
- Filter orders: 15M → 5M
- Join orders-customer: 5M × (300K/1.5M) = 1M
- Same final result, but builds larger hash table (5M vs 300K)
- WORSE than current order

Option 2: lineitem → orders → customer
- Filter lineitem: 60M → 40M
- Would require building hash table on 40M rows
- MUCH WORSE than current order

Option 3: lineitem → customer → orders
- Not semantically valid (lineitem doesn't join directly with customer)
- Would require Cartesian product

CONCLUSION:
-----------

The current join order (customer → orders → lineitem) is OPTIMAL:
1. Smallest table with most selective filter is processed first (customer)
2. Build sides are correctly assigned to smaller relations (customer, then orders)
3. Probe side is correctly assigned to largest relation (lineitem)
4. Intermediate result sizes are minimized at each step

RECOMMENDATION FOR FURTHER OPTIMIZATION:
---------------------------------------

The join order is already optimal. The performance bottleneck identified in
recommendation #3 is about the JOIN EXECUTION STRATEGY (partitioned parallel
hash join), not the join order itself. This optimization should be implemented
by the Execution Optimizer agent, not the Join Order Optimizer agent.

The current implementation issues are:
1. Hash table builds are single-threaded (parallelism issue)
2. Concurrent hash probes contend on shared unordered_map (thread safety issue)
3. No partitioning to improve cache locality (execution strategy issue)

These are outside the scope of join order optimization.
