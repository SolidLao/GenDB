STORAGE DESIGN DECISIONS:
  1. BINARY COLUMNAR FORMAT
     - Eliminates text parsing overhead
     - Enables mmap-based zero-copy access
     - One .bin file per column, ~8.2 GB total

  2. TYPE ENCODING
     - DECIMAL: int64_t with scale_factor=100 (no IEEE 754 errors)
     - DATE: int32_t (days since epoch 1970-01-01)
     - INTEGERS: int32_t
     - LOW-CARDINALITY STRINGS: uint8_t with dictionary encoding
     - HIGH-CARDINALITY STRINGS: length-prefixed binary

  3. DICTIONARY ENCODING (5 columns)
     Applied to:
       - l_returnflag (3 values: N, R, A)
       - l_linestatus (2 values: F, O)
       - l_shipinstruct (4 values)
       - l_shipmode (5 values)
       - c_mktsegment (5 values)
       - o_orderstatus (3 values)
       - o_orderpriority (5 values)
       - p_mfgr (5 values)
       - p_brand (25 values)
       - p_container (10 values)
     Reduces size by 75-90% for these columns

  4. SORT ORDER
     - lineitem: sorted by l_shipdate
       → Enables zone map effectiveness for range queries (Q1, Q3, Q6)
       → Improves cache locality for date-filtered scans
     - orders: sorted by o_orderdate
       → Beneficial for join ordering

  5. INDEXES
     Zone Maps (for range queries):
       - lineitem.l_shipdate: 600 blocks × 100K rows
         Predicates: l_shipdate <= '1998-09-01', l_shipdate >= '1994-01-01'
         Expected skip rate: 50-80% of blocks for typical Q1/Q3/Q6 queries

     Hash Indexes (for point lookups and joins):
       - orders.o_orderkey (15M keys)
       - orders.o_custkey (999K unique keys)
       - customer.c_custkey (1.5M keys)
       - supplier.s_suppkey (100K keys)
       - part.p_partkey (2M keys)
       Used for fast hash joins in Q3 (lineitem ← orders ← customer)
