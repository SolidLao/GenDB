# Query Implementation Strategies (Iteration 1)

## Per-Query Strategy

### Q1: Pricing Summary Report (Simple Aggregation)
**Implementation Approach:**
- Single-table scan of lineitem with date filter → aggregation → trivial sort
- No joins, minimal grouping (3-4 groups: returnflag × linestatus)

**Parquet Predicate Pushdown:**
- Push l_shipdate <= '1998-09-02' to Parquet reader
- CRITICAL: lineitem.parquet is sorted by l_shipdate (verified: RG 0 = 1992-01-02 to 1992-02-01, RG 599 = 1998-11-03 to 1998-12-01)
- Row group pruning: skip row groups where min(l_shipdate) > '1998-09-02'
- Expected: skip ~2% of 600 row groups (last ~12 groups), ~1.2M rows

**Column Projection:**
- Read only 8 of 16 columns: l_returnflag, l_linestatus, l_quantity, l_extendedprice, l_discount, l_tax, l_shipdate, (l_orderkey for row count)
- I/O reduction: ~50% bandwidth savings

**Data Types to Handle:**
- Dates: l_shipdate (date32[day]) → convert to comparable format for filtering
- Decimals: l_quantity, l_extendedprice, l_discount, l_tax (decimal128(15,2)) → convert to double for arithmetic
- Strings: l_returnflag, l_linestatus → dictionary-encoded, use Arrow compute kernels for grouping

**Join Strategy:** N/A (single table)

**Aggregation Strategy:**
- Hash aggregation on (l_returnflag, l_linestatus) → ~4 groups, single-pass
- Compute 6 aggregates: SUM(qty), SUM(price), SUM(disc_price), SUM(charge), AVG(qty), AVG(price), AVG(disc), COUNT(*)
- Use Arrow compute kernels: arrow::compute::Sum, arrow::compute::Mean
- Fuse scan → filter → project → aggregate into single pipeline over Arrow RecordBatches

**Parallelism:**
- Partition lineitem row groups across threads (600 row groups → 8 threads = 75 RG/thread)
- Each thread: local hash table for aggregation, merge at end
- Read row groups in parallel using Arrow's async Parquet reader

**Critical Optimizations:**
1. Parquet row group pruning on l_shipdate (sorted column)
2. Column projection (50% I/O reduction)
3. Parallel row group scanning (morsel-driven parallelism)
4. Single-pass fused scan/filter/aggregate pipeline


### Q3: Shipping Priority Query (Multi-table Join)
**Implementation Approach:**
- Hash join: customer ⋈ orders ⋈ lineitem, then aggregate, then top-K sort

**Join Order & Build/Probe Selection:**
1. **First join: customer ⋈ orders**
   - Build: customer (1.5M rows, filter c_mktsegment='BUILDING' → ~300K rows)
   - Probe: orders (15M rows, filter o_orderdate < '1995-03-15')
   - Join key: c_custkey = o_custkey (PK-FK, 1:N)
   - Expected output: ~300K customers × 10 orders/customer = ~3M qualifying order rows

2. **Second join: (customer⋈orders) ⋈ lineitem**
   - Build: result from first join (~3M order rows, project only o_orderkey, o_orderdate, o_shippriority)
   - Probe: lineitem (60M rows, filter l_shipdate > '1995-03-15')
   - Join key: o_orderkey = l_orderkey (PK-FK, 1:N)
   - Expected output: ~3M orders × 4 line items/order = ~12M rows

**Parquet Predicate Pushdown:**
- **customer:** Push c_mktsegment = 'BUILDING' (dictionary-encoded string equality)
  - No row group pruning benefit (uniformly distributed across row groups)
- **orders:** Push o_orderdate < '1995-03-15'
  - Need to check if o_orderdate is sorted (likely sorted by o_orderkey, not o_orderdate)
  - Minimal row group pruning expected
- **lineitem:** Push l_shipdate > '1995-03-15' (CRITICAL)
  - Row group pruning: skip row groups where max(l_shipdate) <= '1995-03-15'
  - Expected: skip first ~38% of row groups (230 out of 600), ~23M rows not scanned

**Column Projection:**
- **customer:** c_custkey, c_mktsegment (2 of 8 columns) → 75% I/O reduction
- **orders:** o_orderkey, o_custkey, o_orderdate, o_shippriority (4 of 9 columns) → 56% I/O reduction
- **lineitem:** l_orderkey, l_extendedprice, l_discount, l_shipdate (4 of 16 columns) → 75% I/O reduction

**Data Types to Handle:**
- Dates: o_orderdate (date32), l_shipdate (date32) → comparable format
- Decimals: l_extendedprice, l_discount (decimal128(15,2)) → double for revenue calculation
- Integers: all keys (int32) → use as-is for hash join

**Aggregation Strategy:**
- GROUP BY l_orderkey, o_orderdate, o_shippriority
- Expected groups: ~500K (one per qualifying order)
- Aggregate: SUM(l_extendedprice * (1 - l_discount)) as revenue
- Hash aggregation, can pipeline with second join (aggregate as join produces results)

**Top-K Strategy:**
- ORDER BY revenue DESC, o_orderdate LIMIT 10
- Use partial sort (heap-based top-K) to avoid full sort of ~500K groups
- Only materialize top 10 results

**Parallelism:**
- Build hash tables (customer, orders) in parallel
- Parallel lineitem row group scanning (362 qualifying RGs after pruning → 8 threads)
- Partition aggregation by l_orderkey hash (thread-local hash tables, merge at end)

**Critical Optimizations:**
1. Parquet row group pruning on lineitem.l_shipdate (skip 38% of data)
2. Column projection (75% I/O reduction on lineitem, largest table)
3. Build hash tables on smaller sides (customer, then join result)
4. Parallel row group scanning of lineitem (largest scan)
5. Top-K optimization (heap instead of full sort)
6. Pipeline join → aggregate (no materialization between stages)


### Q6: Forecasting Revenue Change (Highly Selective Filter)
**Implementation Approach:**
- Single-table scan of lineitem with 3 filters → single aggregate (no grouping)
- Filters: l_shipdate >= '1994-01-01' AND < '1995-01-01', l_discount BETWEEN 0.05 AND 0.07, l_quantity < 24
- Aggregate: SUM(l_extendedprice * l_discount)

**Parquet Predicate Pushdown:**
- Push l_shipdate range filter ['1994-01-01', '1995-01-01') to Parquet reader
- Row group pruning: only read row groups with overlapping l_shipdate range
- Expected: read only ~100 out of 600 row groups (17% of data, ~10M rows)
- Cannot push l_discount or l_quantity filters (Parquet doesn't track min/max stats well for these)

**Column Projection:**
- Read only 4 of 16 columns: l_shipdate, l_discount, l_quantity, l_extendedprice
- I/O reduction: **75% bandwidth savings** (best of all queries)

**Data Types to Handle:**
- Date: l_shipdate (date32[day]) → convert for range comparison
- Decimals: l_quantity, l_extendedprice, l_discount (decimal128(15,2)) → double for arithmetic
- Result: single double value (revenue)

**Join Strategy:** N/A (single table)

**Filter Strategy:**
- Apply filters in selectivity order:
  1. l_shipdate range (applied at Parquet level, ~17% selectivity)
  2. l_discount BETWEEN 0.05 AND 0.07 (~20% of remaining, ~2% overall)
  3. l_quantity < 24 (~50% of remaining, ~1% overall)
- Use Arrow compute kernels for vectorized filtering: arrow::compute::Filter
- Fuse all filters into single pass over RecordBatches

**Aggregation Strategy:**
- Single accumulator: sum of (l_extendedprice * l_discount) for qualifying rows
- No hash table needed (no grouping)
- Compute revenue expression during filter pass (fused filter+compute)

**Parallelism:**
- Partition qualifying row groups (~100 RGs) across threads → 8 threads = 12-13 RG/thread
- Each thread: local sum accumulator, reduce at end (trivial merge)
- Parallel Parquet row group reading with column projection

**Critical Optimizations:**
1. **Parquet row group pruning on l_shipdate (skip 83% of data!)**
2. **Column projection (75% I/O reduction on remaining data)**
3. Parallel row group scanning (100 RGs in parallel)
4. Fused filter pipeline (single pass: scan → filter → compute → aggregate)
5. Filter ordering by selectivity (date → discount → quantity)
6. Vectorized filtering with Arrow compute kernels


## Shared Observations

### Parquet File Layout (Critical for all queries)
- **lineitem.parquet:** 60M rows, 600 row groups (~100K rows/RG), **sorted by l_shipdate**
  - Row group 0: l_shipdate [1992-01-02, 1992-02-01]
  - Row group 599: l_shipdate [1998-11-03, 1998-12-01]
  - Enables row group pruning for ALL three queries (Q1, Q3, Q6)
- **orders.parquet:** 15M rows, 150 row groups (~100K rows/RG)
- **customer.parquet:** 1.5M rows, 15 row groups (~100K rows/RG)

### Cross-Query Optimization Opportunities
1. **l_shipdate row group pruning is critical for all queries:**
   - Q1: prune last 2% of row groups (dates > '1998-09-02')
   - Q3: prune first 38% of row groups (dates <= '1995-03-15')
   - Q6: read only middle 17% of row groups (dates in 1994)

2. **Column projection benefits all queries (50-75% I/O reduction):**
   - Must read only necessary columns from Parquet
   - Use arrow::parquet::ParquetFileReader with column indices

3. **Decimal → double conversion strategy:**
   - All queries compute arithmetic on decimal128(15,2) columns
   - Convert to double early (at scan level) to avoid repeated conversions
   - Trade-off: precision vs. performance (acceptable for TPC-H)

4. **Parallel row group scanning pattern:**
   - All queries benefit from multi-threaded row group reads
   - Partition row groups across threads (morsel-driven parallelism)
   - Each thread processes subset of row groups independently

5. **Arrow compute kernel usage:**
   - Use arrow::compute::Filter for vectorized filtering (Q1, Q6)
   - Use arrow::compute::Sum, Mean for aggregations (Q1)
   - Use arrow::compute::Take for top-K extraction (Q3)

6. **Dictionary-encoded column handling:**
   - lineitem: l_returnflag, l_linestatus (Q1) → use dictionary indices for grouping
   - customer: c_mktsegment (Q3) → use dictionary-encoded equality comparison
   - Faster than string comparisons

### Implementation Anti-Patterns to Avoid
1. **Do NOT read full Parquet files into memory** → use streaming row group reads
2. **Do NOT materialize intermediate join results** → pipeline joins with aggregation
3. **Do NOT use single-threaded execution** → parallel row group processing is critical
4. **Do NOT skip column projection** → 50-75% I/O waste
5. **Do NOT ignore Parquet row group statistics** → date-based pruning is free performance


## Summary
**Primary strategy:** Leverage Parquet's sorted l_shipdate column for aggressive row group pruning (2-83% of data skipped per query), combine with column projection (50-75% I/O reduction) and parallel row group scanning. Build hash joins on smaller sides (Q3), fuse scan/filter/aggregate pipelines (Q1, Q6), and use Arrow compute kernels for vectorized execution.
