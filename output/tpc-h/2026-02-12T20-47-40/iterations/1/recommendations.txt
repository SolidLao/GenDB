# Query Implementation Strategies (Iteration 1)

## Per-Query Strategy

### Q1: Pricing Summary Report
**Operators**: ScanWithDateFilter → HashAgg
**Parquet Strategy**:
- Use ScanWithDateFilter with l_shipdate <= '1998-09-02' (date32 = 10471)
- Row group pruning: lineitem is sorted by l_shipdate (verified: RG 0 starts 1992-01-02, RG 1199 ends 1998-12-01)
- Expected pruning: minimal (97% of data passes), but eliminates ~17 row groups from Nov-Dec 1998
**Column Projection**:
- Read 10 of 16 columns: l_shipdate, l_returnflag, l_linestatus, l_quantity, l_extendedprice, l_discount, l_tax
- Skip: l_orderkey, l_partkey, l_suppkey, l_linenumber, l_commitdate, l_receiptdate, l_shipinstruct, l_shipmode, l_comment
- I/O reduction: 38% fewer columns
**Aggregation**:
- HashAgg with GROUP BY (l_returnflag, l_linestatus) → ~4 groups
- Aggregates: SUM(qty), SUM(extendedprice), SUM(disc_price), SUM(charge), AVG(qty), AVG(price), AVG(discount), COUNT(*)
**Arrow Compute**:
- Use arrow::compute::greater_equal for date filtering after row group pruning
- Aggregation currently uses hash table (no Arrow compute kernels)
**Expected Performance**: Fast scan (good row group size ~50K rows), minimal aggregation overhead (4 groups)

### Q3: Shipping Priority
**Operators**: ScanWithDateFilter (customer) → ScanWithDateFilter (orders) → HashJoinInt32 → ScanWithDateFilter (lineitem) → HashJoinInt32 → HashAgg → Sort/Limit
**Join Order & Build/Probe**:
1. Build hash table from filtered customer (c_mktsegment='BUILDING', ~30K rows, 20% selectivity)
2. Probe with filtered orders (o_orderdate < '1995-03-15', ~500K rows before join, 33% selectivity)
   - Result: ~100K orders (500K × 20% customer selectivity)
3. Build hash table from customer⋈orders result (~100K rows)
4. Probe with filtered lineitem (l_shipdate > '1995-03-15', ~4M rows, 67% selectivity)
   - Result: ~300K lineitem rows (4M × 100K/1.5M orders × 4 lines/order)
**Parquet Strategy**:
- customer: Full scan with c_mktsegment filter (15 row groups, small table ~128MB)
  - No predicate pushdown (c_mktsegment not sorted), apply filter in-memory
- orders: ScanWithDateFilter on o_orderdate < '1995-03-15' (date32 < 9204)
  - Sorted by o_orderdate (verified: RG 0 starts 1992-01-01, RG 199 ends 1998-08-02)
  - Row group pruning: skip ~67% of row groups (keep only RG 0-66 out of 200)
- lineitem: ScanWithDateFilter on l_shipdate > '1995-03-15' (date32 > 9204)
  - Sorted by l_shipdate (verified range 1992-01-02 to 1998-12-01)
  - Row group pruning: skip ~33% of early row groups (skip RG 0-400, keep RG 401-1199)
**Column Projection**:
- customer: c_custkey, c_mktsegment (2 of 8 columns, 75% pruning)
- orders: o_orderkey, o_custkey, o_orderdate, o_shippriority (4 of 9 columns, 56% pruning)
- lineitem: l_orderkey, l_extendedprice, l_discount, l_shipdate (4 of 16 columns, 75% pruning)
**Aggregation**:
- HashAgg GROUP BY (l_orderkey, o_orderdate, o_shippriority) → ~10K groups
- Aggregate: SUM(l_extendedprice * (1 - l_discount)) as revenue
- Post-aggregation: ORDER BY revenue DESC, o_orderdate ASC, LIMIT 10
**Arrow Compute**:
- Use arrow::compute for c_mktsegment equality filter
- Use arrow::compute::greater for l_shipdate > filter
- Use arrow::compute::less for o_orderdate < filter
- Compute revenue expression: multiply + subtract using Arrow kernels before aggregation
**Expected Performance**: Join-heavy workload, benefits from sorted date columns (major I/O savings on orders/lineitem)

### Q6: Forecasting Revenue Change
**Operators**: ScanWithDateFilter → Filter (additional predicates) → Aggregate (scalar)
**Parquet Strategy**:
- Use ScanWithDateFilter with l_shipdate range [1994-01-01, 1995-01-01) (date32 range [8766, 9131))
- Row group pruning: lineitem sorted by l_shipdate enables aggressive pruning
  - Only read row groups overlapping 1994 (roughly RG 300-500 out of 1200)
  - Expected pruning: skip ~83% of row groups, read only ~200 row groups
- This is the HIGHEST IMPACT optimization for Q6 (17% selectivity → ~1M rows from ~60M)
**Column Projection**:
- Read 4 of 16 columns: l_shipdate, l_quantity, l_discount, l_extendedprice
- Skip: 12 columns (l_orderkey, l_partkey, l_suppkey, l_linenumber, l_tax, l_returnflag, l_linestatus, l_commitdate, l_receiptdate, l_shipinstruct, l_shipmode, l_comment)
- I/O reduction: 75% fewer columns (EXCELLENT pruning opportunity)
**Additional Filters** (applied in-memory after row group pruning):
- l_discount BETWEEN 0.05 AND 0.07 (~20% selectivity on decimal128 column)
- l_quantity < 24 (~50% selectivity)
- Combined in-memory selectivity: ~10% of rows from selected row groups (~100K final rows)
**Aggregation**:
- Single scalar aggregate: SUM(l_extendedprice * l_discount)
- No GROUP BY, single result row
**Arrow Compute**:
- Use arrow::compute for range checks: greater_equal, less (date), and_kleene (combine predicates)
- Use arrow::compute for l_discount BETWEEN: greater_equal + less_equal + and_kleene
- Use arrow::compute for l_quantity <: less
- Combine all filters with and_kleene, then arrow::compute::filter
- Use arrow::compute::multiply for l_extendedprice * l_discount before SUM
**Expected Performance**: Extremely fast due to aggressive row group pruning (83% skipped) + column projection (75% fewer columns)

## Shared Observations

### Parquet Layout (Already Optimized)
- lineitem: sorted by l_shipdate, 1200 row groups, ~50K rows/group → excellent for date range queries
- orders: sorted by o_orderdate, 200 row groups, ~75K rows/group → good for date range pruning
- Row group statistics are available and accurate (verified min/max values are monotonic)

### Critical Success Factors
1. **Exploit sorted columns**: Q3 and Q6 benefit massively from l_shipdate and o_orderdate sorting
2. **Column projection**: Q6 (75% pruning), Q3 (56-75% per table), Q1 (38% pruning)
3. **Row group pruning via ScanWithDateFilter**:
   - Q6: skip 83% of lineitem row groups (1000/1200)
   - Q3: skip 67% of orders row groups (134/200)
   - Q3: skip 33% of lineitem row groups (400/1200)
4. **Join order**: Build hash tables on smaller filtered sides (customer → orders, then combined → lineitem)
5. **Arrow compute kernels**: Use for all filters, comparisons, and arithmetic to leverage vectorization

### Implementation Priorities
1. Implement Q6 first (simplest, highest row group pruning benefit, validates ScanWithDateFilter)
2. Implement Q1 next (single-table aggregation, validates HashAgg with low cardinality groups)
3. Implement Q3 last (most complex, multi-way joins, validates HashJoinInt32 and join ordering)

### Decimal Handling
- TPC-H uses decimal128(15,2) for prices, discounts, quantities
- Cast to float64 for arithmetic operations in Arrow compute
- Handle conversion in ArrayValueToDouble (already implemented in hash_agg.h)

### Filter Strategy Pattern
All queries follow: Parquet predicate pushdown (row group pruning) → Arrow compute filtering (in-memory) → operators

## Summary
Leverage sorted l_shipdate/o_orderdate for aggressive row group pruning (Q6: 83%, Q3 orders: 67%, Q3 lineitem: 33%), maximize column projection (Q6: 75%, Q3: 56-75%), use HashJoinInt32 with smaller-side builds, and apply Arrow compute kernels for all filters and arithmetic.
