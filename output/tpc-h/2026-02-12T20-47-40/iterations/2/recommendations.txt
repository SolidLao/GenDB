## Critical Fixes (Blocking All Execution)

### [CRITICAL] decimal128 Type Handling
**Issue:** hash_agg.h:203 throws "Unsupported aggregate type: decimal128(15, 2)" because ArrayValueToDouble() only handles INT32, INT64, DOUBLE, FLOAT types. TPC-H schema uses decimal128(15,2) for l_quantity, l_extendedprice, l_discount, l_tax columns.

**Fix:** Add DECIMAL128 case to hash_agg.h ArrayValueToDouble() function:
```cpp
case arrow::Type::DECIMAL128: {
    auto decimal_array = std::static_pointer_cast<arrow::Decimal128Array>(array);
    auto decimal_type = std::static_pointer_cast<arrow::Decimal128Type>(array->type());
    int32_t scale = decimal_type->scale();
    arrow::Decimal128 decimal_val = decimal_array->Value(index);
    // Convert decimal128 to double: divide raw value by 10^scale
    double divisor = std::pow(10.0, scale);
    return static_cast<double>(decimal_val.low_bits()) / divisor;
}
```

**Also fix:** Q1, Q3, Q6 use arrow::compute::CallFunction("multiply", "subtract", "add") on decimal128 arrays. Arrow compute functions may preserve decimal128 types. The code assumes results are float64, but they may be decimal128. Need to:
1. Cast decimal128 columns to float64 BEFORE arithmetic operations
2. OR handle decimal128 in ArrayValueToString() for group-by keys (Q1 groups by l_returnflag, l_linestatus which are strings, so this is OK)

**Recommended approach:** In scan.h or query code, immediately cast decimal128 columns to float64 after reading from Parquet:
```cpp
auto price_arr = arrow::Concatenate(table->GetColumnByName("l_extendedprice")->chunks()).ValueOrDie();
auto cast_result = arrow::compute::CallFunction("cast", {arrow::Datum(price_arr)},
    arrow::compute::CastOptions::Safe(arrow::float64()));
auto price_float64 = cast_result.ValueOrDie().make_array();
```
