# Workload Analysis

## Scale Factor: 10
## Optimization Target: execution_time

## Tables (estimated sizes at SF=10)
- lineitem: ~6,000,000 rows (fact table, ~750MB raw)
  Hot columns: l_shipdate, l_quantity, l_discount, l_extendedprice, l_returnflag, l_linestatus
- orders: ~1,500,000 rows (fact table, ~165MB raw)
  Hot columns: o_orderkey, o_custkey, o_orderdate, o_shippriority
- customer: ~150,000 rows (dimension, ~24MB raw)
  Hot columns: c_custkey, c_mktsegment
- nation: ~25 rows (dimension, <1KB)
- region: ~5 rows (dimension, <1KB)
- supplier: ~10,000 rows (dimension, ~1.4MB raw)
- part: ~200,000 rows (dimension, ~24MB raw)
- partsupp: ~800,000 rows (fact, ~112MB raw)

## Query Patterns

### Q1: Pricing Summary Report (lineitem scan + aggregate)
- Single table scan: lineitem
- Filter: l_shipdate <= '1998-09-02' (~97% of data passes)
- Projection: 10 of 16 columns (38% pruning)
- Aggregation: GROUP BY l_returnflag, l_linestatus (~4 groups expected)
- No joins

### Q3: Shipping Priority (3-way join + aggregate)
- Tables: customer, orders, lineitem
- Joins:
  - customer.c_custkey = orders.o_custkey (PK-FK)
  - orders.o_orderkey = lineitem.l_orderkey (PK-FK)
- Filters:
  - c_mktsegment = 'BUILDING' (~20% selectivity, 30K customers)
  - o_orderdate < '1995-03-15' (~33% selectivity, ~500K orders)
  - l_shipdate > '1995-03-15' (~67% selectivity, ~4M lineitem rows)
- Combined selectivity: ~4-5% of lineitem rows after all filters
- Aggregation: GROUP BY l_orderkey, o_orderdate, o_shippriority (~10K groups expected)
- Top-10 result via ORDER BY + LIMIT

### Q6: Forecasting Revenue Change (lineitem scan + aggregate)
- Single table scan: lineitem
- Filters:
  - l_shipdate >= '1994-01-01' AND < '1995-01-01' (~17% selectivity, ~1M rows)
  - l_discount BETWEEN 0.05 AND 0.07 (~20% selectivity)
  - l_quantity < 24 (~50% selectivity)
- Combined selectivity: ~1.7% of lineitem rows (~100K rows)
- Projection: 4 of 16 columns (75% pruning) ← EXCELLENT column pruning opportunity
- No joins, simple SUM aggregation

## Selective Filters (good candidates for optimization)

1. **l_shipdate range filters** (all 3 queries)
   - Q1: <= '1998-09-02' (low selectivity, but universal filter)
   - Q3: > '1995-03-15' (67% selectivity)
   - Q6: >= '1994-01-01' AND < '1995-01-01' (17% selectivity) ← HIGHLY SELECTIVE
   → Sort lineitem by l_shipdate to enable row group pruning

2. **l_discount BETWEEN** (Q6): ~20% selectivity on numeric column

3. **l_quantity < 24** (Q6): ~50% selectivity

4. **c_mktsegment = 'BUILDING'** (Q3): ~20% selectivity
   → Dictionary encoding beneficial

5. **o_orderdate < '1995-03-15'** (Q3): ~33% selectivity
   → Sort orders by o_orderdate to enable row group pruning

## Join Graph

- customer.c_custkey = orders.o_custkey (PK-FK, 1:N)
  Used in: Q3
  Estimated join selectivity: ~20% (due to c_mktsegment filter)

- orders.o_orderkey = lineitem.l_orderkey (PK-FK, 1:N, avg ~4 lineitems/order)
  Used in: Q3
  Estimated join selectivity: ~33% (due to o_orderdate filter)

## Aggregations

- Q1: GROUP BY l_returnflag, l_linestatus
  Estimated groups: 4 (low cardinality)
  Dictionary encoding will accelerate grouping

- Q3: GROUP BY l_orderkey, o_orderdate, o_shippriority
  Estimated groups: ~10K (after filters)
  Requires hash aggregation

- Q6: Single aggregate (no GROUP BY)

## Column Projection Opportunities

- Q6: Uses only 4 of 16 lineitem columns
  Columns needed: l_shipdate, l_quantity, l_discount, l_extendedprice
  Columns skipped: 12 (75% pruning)
  → Columnar Parquet format provides significant I/O savings

- Q1: Uses 10 of 16 lineitem columns (38% pruning)

- Q3: Uses ~8 lineitem columns, ~4 orders columns, ~2 customer columns

## Parallelism Opportunities

- Q1: Full lineitem scan, highly parallelizable (partition by row groups)
- Q6: Lineitem scan with selective filters, parallelizable (partition by row groups)
- Q3: Join-heavy, parallelize via:
  - Partition customer filter (c_mktsegment)
  - Partition orders filter (o_orderdate)
  - Partition lineitem scan (l_shipdate)
  - Use hash partitioning for joins

## Key Optimization Opportunities

1. **Sort lineitem by l_shipdate**
   - Enables row group pruning for Q1, Q3, Q6 date range filters
   - Q6 can skip ~83% of row groups (reading only 1994 data)
   - Critical for execution time optimization

2. **Smaller row groups for lineitem (50K rows)**
   - Finer-grained pruning on sorted l_shipdate column
   - Trade-off: slightly more metadata, but worth it for selective Q6

3. **Dictionary encoding for low-cardinality strings**
   - lineitem: l_returnflag (3 values), l_linestatus (2 values), l_shipinstruct (~4 values), l_shipmode (~7 values)
   - customer: c_mktsegment (5 values) ← accelerates Q3 filter
   - orders: o_orderstatus (3 values), o_orderpriority (5 values)
   - Reduces storage + accelerates equality filters and GROUP BY

4. **Sort orders by o_orderdate**
   - Enables row group pruning for Q3 filter (o_orderdate < '1995-03-15')
   - Can skip ~67% of row groups

5. **Compression: Snappy (default)**
   - Good balance of speed and compression ratio
   - Optimizes for execution time target

## Workload Characterization

- **Read-heavy analytical workload** (3 read-only queries, no writes)
- **Scan-intensive**: All queries scan lineitem (largest table)
- **Filter-first pattern**: Selective predicates before aggregation/joins
- **Low aggregation cardinality**: Small result sets (Q1: 4 rows, Q3: 10 rows, Q6: 1 row)
- **Column-sparse access**: Q6 demonstrates high column pruning potential
- **Date range filtering dominance**: l_shipdate is the critical optimization axis
