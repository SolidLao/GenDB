BOTTLENECK ANALYSIS: WHY SEMI_JOIN_BUILD STILL DOMINATES
════════════════════════════════════════════════════════════════════════════

Current (Iter 1): 632.68 ms for processing 59.986M lineitem rows

What's Happening:
  1. Load 3 columns (l_orderkey, l_commitdate, l_receiptdate)     ~200 ms
  2. Filter: l_commitdate < l_receiptdate (59M → 13.753M match)   ~150 ms
  3. Hash table insertions (13.753M insertions with probing)       ~282 ms
                                                             ───────────
                                                Total:          ~632 ms


Memory Bandwidth Analysis:
───────────────────────────────────────────────────────────────────────────
Data Volume: 59.986M rows × 12 bytes/row = ~720 MB
CPU Memory Bandwidth: ~100 GB/s (typical modern CPU)
Theoretical Minimum: 720 MB ÷ 100 GB/s = 7.2 ms

Current Reality: 632.68 ms
Multiplier: 632.68 / 7.2 = 88x of theoretical minimum

Why the Gap?
  - Memory prefetching overhead (TLB misses, L3 eviction)
  - Branch prediction misses on filter
  - Hash function computation cost (~5 cycles each)
  - Linear probing overhead (avg 1.5 probes per insert)
  - Cache line traffic (memory writes)

Realistic Expected Minimum: 300-500 ms for this architectural approach
Current Performance: 632.68 ms (within 1.3x of expected)
Conclusion: Very close to optimal for full-table-scan approach

TIMING BREAKDOWN ESTIMATES FOR ITER 1 semi_join_build
════════════════════════════════════════════════════════════════════════════

Phase 1: Load 3 columns into memory
  - Cold cache, TLB misses, page faults
  - 720 MB ÷ 100 GB/s × (1 + 0.5 overhead) = ~10.8 ms optimal
  - Actual with stalls: ~50-100 ms estimated

Phase 2: Sequential filter scan with memory writes
  - 59.986M iterations
  - Per-iteration cost: hash load + comparison + branch + conditional jump
  - Memory prefetch helps here (sequential access)
  - Estimated: ~100-150 ms

Phase 3: Hash table insertion (13.753M operations)
  - Hash function: ~5 cycles
  - Linear probing (avg 1.5 probes): ~10 cycles
  - Memory write: ~20 cycles
  - Per-insert: ~75 cycles
  - At 3 GHz: 13.753M × 75 cycles ÷ 3B cycles/s = ~344 ms
  - But cache effects: could be 250-350 ms actual
  - Estimated: ~280-350 ms (matches observation of 632 total)

Overhead (lock contention, thread sync, etc):
  - Minimal in single-threaded phase
  - Estimated: <10 ms

Total Estimate: 50 + 100 + 315 = 465 ms (vs actual 632 ms)
Difference: Likely due to actual memory bandwidth being lower,
           or worse cache behavior than estimated
