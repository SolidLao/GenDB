# Storage Design

## Hardware
CPU: 64 cores, L3: 44MB, HDD, 376GB RAM

## Format
Binary columnar. One file per column. mmap access. Block size: 8192 rows (44MB L3 / 64 threads / ~10 hot columns ≈ 70KB/thread/block).

## Table Designs

### lineitem (~6M rows, fact table)
Sort by: l_shipdate (all 3 queries filter on date ranges, enables zone-map block skipping)
Dictionary encode: l_returnflag (3 values → uint8_t), l_linestatus (2 values → uint8_t), l_shipmode (7 values → uint8_t), l_shipinstruct (4 values → uint8_t)
Scaled: l_quantity, l_extendedprice, l_discount, l_tax → int64_t (× 100)
Date: l_shipdate, l_commitdate, l_receiptdate → int32_t (days since epoch)
PK index: (l_orderkey, l_linenumber) composite hash index

### orders (~1.5M rows, fact table)
Sort by: o_orderdate (Q3 filters on this, enables block skipping)
Dictionary encode: o_orderstatus (3 values → uint8_t), o_orderpriority (5 values → uint8_t)
Scaled: o_totalprice → int64_t (× 100)
Date: o_orderdate → int32_t
PK index: o_orderkey

### customer (~150K rows, dimension)
Dictionary encode: c_mktsegment (5 values → uint8_t)
Scaled: c_acctbal → int64_t (× 100)
PK index: c_custkey

### partsupp (~800K rows, dimension)
Scaled: ps_supplycost → int64_t (× 100)
PK index: (ps_partkey, ps_suppkey) composite hash index

### part (~200K rows, dimension)
Dictionary encode: p_container (40 values → uint8_t)
Scaled: p_retailprice → int64_t (× 100)
PK index: p_partkey

### supplier (~10K rows, dimension)
Scaled: s_acctbal → int64_t (× 100)
PK index: s_suppkey

### nation (25 rows), region (5 rows)
No special encoding. PK indexes: n_nationkey, r_regionkey.

## Type Mappings
INTEGER → int32_t, DECIMAL(x,y) → int64_t (× 100), DATE → int32_t, CHAR/VARCHAR → std::string (or uint8_t if dictionary-encoded)

## Parallel Ingestion Strategy
- All 8 tables ingested concurrently via std::thread pool
- lineitem (6M rows): mmap input .tbl file, split into 64 chunks by newline boundaries, parse each chunk in parallel into thread-local vectors, merge vectors, parallel sort by l_shipdate using std::sort with execution policy
- orders (1.5M rows): same parallel approach, 64 chunks, parallel sort by o_orderdate
- Medium tables (partsupp 800K, part 200K, customer 150K): 32 chunks each, parallel parse, no sort
- Small tables (supplier 10K, nation 25, region 5): single-thread parse
- Pre-allocation: estimate rows = file_size / 150 bytes (avg row), reserve all vectors
- Batch writes: one fwrite() per column after full construction
