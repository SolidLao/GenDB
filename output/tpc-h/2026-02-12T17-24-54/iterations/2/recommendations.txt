### 3. [cpu_bound] Optimize Q3 join execution
**Target:** Q3 (6425ms → target 100-200ms, ~32x improvement needed)
**Expected:** 10-20x speedup
**Risk:** Medium (requires refactoring join strategy)
**Guidance:**
- **Root cause:** Q3 materializes all join pairs into vectors (line 53: cust_order_matches, line 89: build_keys/build_rows), causing excessive memory allocation and cache misses. With ~300K customer-order matches and ~600K order-lineitem matches, this creates 180M match pairs before aggregation.
- **Solution 1 - Operator fusion (recommended):**
  - Fuse customer→orders→lineitem joins into single pass with direct aggregation
  - Eliminate materialization of intermediate join results
  - Use hash table probe to directly update revenue aggregates
  - Reference: q3.cpp lines 78-101 already use callback-based fusion for orders→lineitem, extend this pattern to include customer filter
- **Solution 2 - Bloom filter for join reduction:**
  - Build bloom filter on filtered customer custkeys
  - Use bloom filter to pre-filter orders before hash join
  - Expected: 80% reduction in orders hash table size (1.5M → 300K)
- **Implementation notes:**
  - Keep parallel execution (lines 92-101 already do per-thread aggregation)
  - Hash table is read-only during probe, no contention
  - Use std::unordered_map::reserve() for hash tables to avoid rehashing

### 4. [io_bound] Add madvise hints for sequential scans
**Target:** Q1, Q6 (I/O bound full table scans)
**Expected:** 1.5-2x speedup on HDD, minimal benefit on SSD/RAM cache
**Risk:** Low (non-invasive kernel hint)
**Guidance:**
- storage/storage.cpp: After mmap() call, add `madvise(addr, length, MADV_SEQUENTIAL | MADV_WILLNEED)`
- Helps kernel prefetch sequential pages for full scans
- Q1 scans 7 columns × ~6M rows × 8 bytes ≈ 336MB sequentially
- Q6 scans 4 columns × ~6M rows × 8 bytes ≈ 192MB sequentially
- **Caution:** Don't apply to random-access columns (join keys, filtered columns)

### 5. [index] Zone-map metadata for block skipping (future iteration)
**Target:** Q6 (could achieve 2-3x speedup), Q1/Q3 (10-30% speedup)
**Expected:** 2-3x for Q6, 1.1-1.3x for Q1/Q3
**Risk:** High (requires storage format change, but storage is never rebuilt per task constraints)
**Guidance:**
- **Note:** This optimization CANNOT be implemented in current iteration because storage is never rebuilt after initial ingestion
- If storage rebuild becomes allowed:
  - During ingest, compute min/max for l_shipdate, l_discount, l_quantity per 8192-row block
  - Store zone-maps in separate metadata file (e.g., lineitem.l_shipdate.zonemap)
  - During query execution, read zone-map and skip blocks where:
    - Q6: l_shipdate block_max < DATE_1994_01_01 OR block_min >= DATE_1995_01_01
    - Q6: l_discount block_max < 5 OR block_min > 7
    - Q6: l_quantity block_min >= 2400
  - Expected block skip rate for Q6: 70-85% (date range is 1/7 of data)
- **Alternative without storage rebuild:** Build runtime zone-map on first query, cache in memory
  - Scan l_shipdate once to build block min/max array
  - Amortized over multiple queries in workload
  - Risk: adds overhead to first query execution

### 6. [cpu_bound] SIMD vectorization for Q6 filters
**Target:** Q6 (currently 22ms, could reach ~10-15ms)
**Expected:** 1.5-2x speedup
**Risk:** Medium (requires AVX2/SSE intrinsics, compiler support)
**Guidance:**
- Q6 has 4 independent predicates on each row, ideal for SIMD
- Current code (lines 43-47) processes 1 row per iteration
- **Vectorized approach:**
  - Process 4-8 rows per iteration using AVX2 (256-bit) or AVX-512 (512-bit)
  - Use `_mm256_load_si256()` to load 4 int64_t values at once
  - Use `_mm256_cmpgt_epi64()`, `_mm256_and_si256()` for parallel comparisons
  - Use `_mm256_movemask_epi8()` to extract match mask
  - Aggregate matches using `_mm256_add_epi64()`
- **Implementation:**
  - Add SIMD version in q6.cpp lines 43-52 with compiler flag check (`#ifdef __AVX2__`)
  - Fall back to scalar loop for remainder rows (row_count % 4)
  - Keep parallel threading (already implemented lines 37-55)
- **Compiler flags:** Add `-mavx2` to Makefile CXXFLAGS
