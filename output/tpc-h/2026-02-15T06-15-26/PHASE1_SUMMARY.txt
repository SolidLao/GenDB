================================================================================
KEY DESIGN DECISIONS & JUSTIFICATIONS
================================================================================

1. BINARY COLUMNAR STORAGE (vs Row-oriented)
   ✓ Column pruning: Q1 scans only 6 of 16 lineitem columns
   ✓ Zero-copy mmap: Eliminates text parsing (10 min → 1 sec for 60M rows)
   ✓ Compression: Dictionary encoding + native binary → 19% reduction
   ✓ Vectorization: Column-major layout enables SIMD filtering

2. SCALED FIXED-POINT DECIMALS (vs IEEE 754 double)
   ✓ Precision: No boundary errors in aggregations (IEEE 754 has rounding)
   ✓ Comparison: Exact decimal comparisons (e.g., 0.05 BETWEEN query)
   ✓ Performance: int64_t operations faster than floating-point
   ✗ Problem: std::from_chars<int32_t> on "0.04" reads only "0"
   ✓ Solution: Parse as double, then convert to int64_t

3. EPOCH-BASED DATE ENCODING (vs YYYYMMDD integers)
   ✓ Range queries: l_shipdate <= date is simple integer comparison
   ✓ Size: 4 bytes per date (not 8 bytes for YYYYMMDD)
   ✓ Zone maps: Effective for "1994 <= year <= 1998" predicates
   ✗ Problem: std::from_chars<int32_t> on "1998-12-01" reads only "1998"
   ✓ Solution: Manual YYYY-MM-DD parsing (loop over years/months/days)

4. DICTIONARY ENCODING FOR LOW-CARDINALITY COLUMNS
   ✓ I/O reduction: 3 codes (uint8_t) vs 1-3 bytes per string
   ✓ SIMD filtering: Vectorized comparison on uint8_t codes
   ✓ Cardinality: l_returnflag=3, l_linestatus=2, c_mktsegment=5
   ✓ Q1 aggregation: GROUP BY l_returnflag uses 3 groups

5. ZONE MAPS ON DATE COLUMNS (vs other columns)
   ✓ Workload: l_shipdate filters in Q1, Q3, Q6
   ✓ Selectivity: "shipdate <= '1998-09-01'" is 98% selective
   ✓ Block skip: Zone map can prune 0-100% of blocks
   ✓ Cost: 9.4 KB metadata for 600 blocks

6. HASH INDEXES ON PK/FK COLUMNS (vs all columns)
   ✓ Join acceleration: Hash probe on o_orderkey, o_custkey, c_custkey
   ✓ Memory: 29M entries × 12 bytes = 348 MB (fits in L3 + DRAM)
   ✓ Workload: Q3 join requires fast lookup
   ✗ Not on o_orderdate: Zone maps sufficient for range predicates

7. SORTED INDEX ON l_orderkey (vs others)
   ✓ Merge join: Pre-sorted input for efficient multi-way joins
   ✓ Range scans: Prefix search on order key range
   ✓ Size: 229 MB permutation (4 bytes × 59M rows)
   ✓ Q3: GROUP BY l_orderkey after join

8. SORT BY l_shipdate (vs no sort)
   ✓ Zone map effectiveness: Clustering by filter column
   ✓ Scan performance: Sequential I/O (HDD optimization)
   ✓ Query Q1: Most selective filter is l_shipdate
