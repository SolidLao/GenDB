                         KEY DESIGN DECISIONS
================================================================================

1. SORTED LAYOUT (lineitem by l_shipdate, orders by o_orderdate)
   → Enables zone map pruning on common date filters
   → Expected 40-85% block skip on range predicates

2. DICTIONARY ENCODING (10 columns)
   → 62% column compression for low-cardinality strings
   → Fast filtering on encoded int32_t IDs

3. HASH MULTI-VALUE INDEXES (6 join columns)
   → O(1) lookup for fact-to-dimension joins
   → Sort-based construction avoids std::unordered_map overhead
   → Total index size: 654 MB (vs. 5.6 GB data = 11.7% overhead)

4. ZONE MAPS (3 date/segment columns)
   → Negligible storage (11 KB) with high pruning value
   → Block size: 100K rows (fits in L3 cache for 2 cores)

5. DECIMAL AS INT64_T SCALED (13 columns)
   → Exact arithmetic (no IEEE 754 errors)
   → Native CPU operations on integers (2-10x faster than FP)
