# Q3 I/O Optimization Summary

## Hardware Detected
- Disk Type: HDD (rota=1)
- CPU Cores: 64
- L3 Cache: 44MB
- Memory: 311GB
- Storage Format: Binary Columnar (mmap-based)

## I/O Optimizations Applied (Phase 2)

### 1. Adaptive madvise() Hints for HDD Workload
- **Customer table** (hash join lookups): Changed from MADV_SEQUENTIAL to MADV_RANDOM
  - Justification: Hash join performs random lookups by customer key, not sequential scans
  - HDD Benefit: Prevents speculative read-ahead, reduces unnecessary I/O thrashing
  
- **Orders table** (hash join lookups): Changed from MADV_SEQUENTIAL to MADV_RANDOM
  - Justification: Hash join probes orders by orderkey, not sequential scans
  - HDD Benefit: Avoids read-ahead on non-sequential access patterns
  
- **Lineitem table** (full scan): Kept MADV_SEQUENTIAL
  - Justification: Full scan of ~60M rows with date and orderkey filters
  - HDD Benefit: Enables kernel read-ahead optimization for sequential I/O

### 2. Column Pruning (I/O Reduction)
- Only 9 of 16 lineitem columns are loaded via mmap:
  - l_orderkey, l_extendedprice, l_discount, l_shipdate (filtered columns)
  - o_custkey, o_orderkey, o_orderdate, o_shippriority (join/output columns)
  - c_mktsegment (filter column)
- Unused columns: l_partkey, l_suppkey, l_linenumber, l_quantity, l_tax, l_commitdate, l_receiptdate, l_returnflag, l_linestatus, l_shipinstruct, l_shipmode, l_comment
- **I/O Savings**: ~43% bandwidth reduction per lineitem scan (9/16 columns loaded)

### 3. Morsel-Level Prefetching for HDD
- Added MADV_WILLNEED prefetching for lineitem columns at morsel boundaries
- Morsel size: 8000 rows, tuned for 64 cores and 44MB L3 cache
- Prefetch distance: 256 rows ahead
- **Benefit**: Kernel pages in critical data before morsel processing begins, reducing stalls

### 4. Parallelization (from iter1, enhanced for I/O)
- 64 threads with morsel-driven execution
- Each thread maintains independent hash aggregation table (reduces lock contention)
- Aggregate merge after all threads complete
- **I/O Benefit**: Parallel threads can issue multiple prefetch requests concurrently, improving HDD scheduling

## Remaining Blocker: Semantic Correctness

The code produces **incorrect results** due to date encoding mismatch (identified in iter_0_evaluation.json).

### Root Cause
- Storage encodes dates as year-only integers (1992-1998)
- SQL query requires comparison against specific dates (1995-03-15)
- With year-only encoding, cannot distinguish between 1995-01-01 and 1995-12-31
- Result: Different orderkeys selected than expected (e.g., expected 4791171 but got 30417318)

### Critical Fixes Needed (Outside I/O Optimizer Scope)
Per evaluation recommendations, the following must be fixed in a subsequent iteration:
1. **Fix date filtering logic**: Either decode year-only values to full dates or use a different date encoding
2. **Fix date output formatting**: Convert date integers to proper YYYY-MM-DD format
3. **Verify join logic**: Cross-check intermediate cardinalities with reference database

## I/O Optimization Techniques Available but Not Yet Applied

### Zone Maps (Block Skipping) - Not Applied Yet
- Storage design specifies l_shipdate_zonemap index
- Could skip lineitem blocks whose l_shipdate range doesn't match filter predicates
- **Potential benefit**: Skip 30-50% of blocks, reduce I/O by 0.3-0.5 seconds
- **Blocker**: Date encoding bug makes zone map effectiveness unclear

### Parallel Column Reads - Not Applied Yet
- HDD workload could benefit from reading independent columns in parallel
- Requires thread pool for concurrent disk I/O
- **Potential benefit**: 20-30% reduction in lineitem scan time if columns are on different disks
- **Deferred**: Pending semantic correctness fix
