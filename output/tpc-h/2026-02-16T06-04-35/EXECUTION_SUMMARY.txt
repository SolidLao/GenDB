================================================================================
DESIGN DECISIONS
================================================================================

1. BINARY COLUMNAR STORAGE
   Rationale: Eliminates text parsing (10s of minutes → <3 minutes) for 60M-row fact table.
   Zero-copy mmap I/O; only parse indexed columns or prefiltered ranges.

2. DATE ENCODING (int32_t epoch days)
   Formula: Cumulative days from 1970-01-01 using arithmetic (no mktime).
   Self-test: parse_date("1970-01-01") = 0 ensures no off-by-one errors.
   Example: 1995-03-15 = 9204 days (not 19950315).

3. DECIMAL ENCODING (int64_t with scale_factor=100)
   Avoids IEEE 754 floating-point rounding errors.
   Parse: string → double → multiply by 100 → round → int64_t.
   Pitfall avoided: Using std::from_chars<int32_t> on "0.04" (reads "0").

4. DICTIONARY ENCODING (int8_t/int16_t)
   Low-cardinality columns: returnflag (3), linestatus (2), mktsegment (5), etc.
   Space savings ~80% vs. inline strings (e.g., 3 bytes → 1 byte).
   Dictionary files: lineitem/l_returnflag_dict.txt (code=value mapping).

5. ZONE MAPS (block-level min/max)
   Block size: 200K rows (lineitem), 150K rows (orders).
   Rationale: Prunes entire blocks for range predicates (e.g., l_shipdate <= DATE '1998-09-01').
   Lineitem: 300 zones (59.9M / 200K), compact binary format ~3.6KB.

6. HASH MULTI-VALUE INDEXES (join columns)
   Design: Two-array approach (positions array + hash table).
   Why: Many-to-many joins (e.g., l_orderkey appears ~4 times per row).
   One hash entry per unique key (not per row) → small hash table.
   Load factor 0.5–0.6 → fast probing with few collisions.
   Example: lineitem_l_orderkey: 15M unique keys, 573MB index.

7. PARALLELISM
   Ingestion: Sequential per table (no nested parallelism), but columnar writes are fast.
   Index building: OpenMP for parallel zone map construction, parallel histogram.
   64 cores × 44MB L3 → morsel size ~170K rows for CPU-local processing.
