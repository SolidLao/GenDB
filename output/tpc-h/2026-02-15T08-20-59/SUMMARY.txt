KEY DESIGN DECISIONS
--------------------

1. DECIMAL HANDLING
   - Stored as int64_t with scale_factor=100 (NOT double/float)
   - Ensures precision (e.g., 0.04 → 4, exactly)
   - Avoids IEEE 754 boundary errors
   - Parsing: double → scale → round → int64_t

2. DATE HANDLING
   - Stored as int32_t days since 1970-01-01
   - Manual YYYY-MM-DD parsing (leap-year aware arithmetic)
   - Example: 1998-12-01 = 10592 days (NOT "19981201")
   - Verified: min=8037 (1992-02-18), max=10562 (1998-11-13) ✓

3. DICTIONARY ENCODING
   - Lineitem: l_returnflag (3 values), l_linestatus (2 values), l_shipinstruct, l_shipmode
   - Orders: o_orderstatus, o_orderpriority, o_clerk
   - Customer: c_mktsegment (5 values)
   - Compression: ~5–10× for CHAR columns
   - Dictionary files: Text format (code=value), loaded at runtime

4. SORT ORDER
   - Lineitem: Sorted by l_shipdate (high-cardinality, frequently filtered)
   - Orders: Sorted by o_orderdate (Q3 date range)
   - Customer: Unsorted (small table, no sort benefit)
   - Benefit: Zone maps on sort keys extremely effective

5. BLOCK SIZE
   - 131,072 rows per block (128K)
   - Rationale: ~1 MB per block for typical columns (4–8 bytes/row)
   - Fits within 4 KB page cache efficiently
   - 458 blocks for lineitem (60M rows)
