TPC-H WORKLOAD ANALYSIS (SF=10, Queries: Q1, Q3, Q6)
======================================================

OPTIMIZATION TARGET: execution_time

TABLE CHARACTERISTICS
---------------------

1. lineitem (FACT TABLE - ~60M rows at SF=10)
   Role: Primary fact table, dominates all queries
   Hot columns: l_shipdate, l_orderkey, l_extendedprice, l_discount, l_quantity,
                l_returnflag, l_linestatus, l_partkey, l_suppkey

2. orders (FACT TABLE - ~15M rows at SF=10)
   Role: Order header table, joins with lineitem and customer
   Hot columns: o_orderkey, o_custkey, o_orderdate, o_shippriority

3. customer (DIMENSION - ~1.5M rows at SF=10)
   Role: Customer dimension, filtered by mktsegment
   Hot columns: c_custkey, c_mktsegment

4. supplier (DIMENSION - ~100K rows at SF=10)
   Role: Supplier dimension
   Hot columns: s_suppkey, s_nationkey

5. part (DIMENSION - ~2M rows at SF=10)
   Role: Part dimension
   Hot columns: p_partkey

6. partsupp (FACT TABLE - ~8M rows at SF=10)
   Role: Part-supplier relationship
   Hot columns: ps_partkey, ps_suppkey

7. nation (DIMENSION - 25 rows)
   Role: Tiny lookup table

8. region (DIMENSION - 5 rows)
   Role: Tiny lookup table

JOIN GRAPH
----------

customer.c_custkey = orders.o_custkey
  - 1:N relationship
  - Join selectivity: ~10 orders per customer on average
  - Used in Q3 with selective filter on c_mktsegment

orders.o_orderkey = lineitem.l_orderkey
  - 1:N relationship
  - Join selectivity: ~4 lineitems per order on average
  - Used in Q3, critical join path

lineitem.l_partkey,l_suppkey = partsupp.ps_partkey,ps_suppkey
  - N:1 relationship
  - Not used in provided queries but defined in schema

supplier.s_nationkey = nation.n_nationkey
nation.n_regionkey = region.r_regionkey
  - Not used in provided queries

SELECTIVE FILTERS
-----------------

Q1: l_shipdate <= '1998-09-02'
  - Estimated selectivity: ~95% (filters out ~5% of late shipments)
  - Column: l_shipdate (DATE)
  - Filter type: Range (<=)

Q3: c_mktsegment = 'BUILDING'
  - Estimated selectivity: ~20% (5 market segments in TPC-H)
  - Column: c_mktsegment (CHAR(10))
  - Filter type: Equality

Q3: o_orderdate < '1995-03-15'
  - Estimated selectivity: ~25-30% (depends on date range in data)
  - Column: o_orderdate (DATE)
  - Filter type: Range (<)

Q3: l_shipdate > '1995-03-15'
  - Estimated selectivity: ~70-75%
  - Column: l_shipdate (DATE)
  - Filter type: Range (>)

Q6: l_shipdate >= '1994-01-01' AND l_shipdate < '1995-01-01'
  - Estimated selectivity: ~14% (1 year out of 7 years typical TPC-H date range)
  - Column: l_shipdate (DATE)
  - Filter type: Range (>=, <)

Q6: l_discount BETWEEN 0.05 AND 0.07
  - Estimated selectivity: ~18% (2 out of 11 discount values)
  - Column: l_discount (DECIMAL)
  - Filter type: Range (BETWEEN)

Q6: l_quantity < 24
  - Estimated selectivity: ~48% (24 out of 50 typical quantity range)
  - Column: l_quantity (DECIMAL)
  - Filter type: Range (<)

AGGREGATIONS
------------

Q1: GROUP BY l_returnflag, l_linestatus
  - Estimated cardinality: ~4 groups (3 returnflags × 2 linestatuses, but not all combinations exist)
  - Low cardinality, in-memory aggregation efficient

Q3: GROUP BY l_orderkey, o_orderdate, o_shippriority
  - Estimated cardinality: ~10 groups (LIMIT 10, but computes across all matching orders)
  - After filters, ~1-5% of orders remain
  - Medium cardinality, requires hash aggregation

COLUMN PROJECTION OPPORTUNITIES
--------------------------------

Q1: Uses 8 of 16 lineitem columns (50% projection)
  - Reads: l_returnflag, l_linestatus, l_quantity, l_extendedprice, l_discount, l_tax, l_shipdate
  - Skips: l_orderkey, l_partkey, l_suppkey, l_linenumber, l_returnflag variants, dates, shipping fields

Q3: Uses 4 of 16 lineitem columns (25% projection)
  - Reads: l_orderkey, l_extendedprice, l_discount, l_shipdate
  - High columnar benefit

Q3: Uses 4 of 9 orders columns (44% projection)
  - Reads: o_orderkey, o_custkey, o_orderdate, o_shippriority

Q3: Uses 2 of 8 customer columns (25% projection)
  - Reads: c_custkey, c_mktsegment

Q6: Uses 4 of 16 lineitem columns (25% projection)
  - Reads: l_shipdate, l_discount, l_quantity, l_extendedprice
  - Excellent columnar benefit

INDEX RECOMMENDATIONS
---------------------

PRIMARY INDEXES (for point lookups and join optimization):

lineitem:
  - l_orderkey: Primary key component, used in joins (Q3)
  - l_shipdate: Most selective filter across all queries (Q1, Q3, Q6)
    Rationale: Sorted index on l_shipdate enables fast range scans for date filters

orders:
  - o_orderkey: Primary key, join key with lineitem
  - o_orderdate: Selective filter in Q3
  - o_custkey: Foreign key, join with customer

customer:
  - c_custkey: Primary key, join key with orders
  - c_mktsegment: Selective equality filter in Q3

part:
  - p_partkey: Primary key (not used in provided queries but likely in full workload)

supplier:
  - s_suppkey: Primary key (not used in provided queries but likely in full workload)

partsupp:
  - ps_partkey, ps_suppkey: Composite primary key (not used in provided queries)

FOREIGN KEY INDEXES (for join acceleration):

lineitem:
  - l_partkey: Foreign key to part
  - l_suppkey: Foreign key to supplier

customer:
  - c_nationkey: Foreign key to nation

supplier:
  - s_nationkey: Foreign key to nation

PARALLELISM OPPORTUNITIES
-------------------------

1. Q1: Single-table scan of lineitem
   - Partition by l_shipdate ranges for parallel scan
   - Each thread processes row groups independently
   - Final aggregation merges 4 groups (low overhead)

2. Q3: Multi-way join
   - Parallel scan of customer (small, can broadcast)
   - Parallel scan of orders with date filter
   - Parallel probe of lineitem
   - Partition hash aggregation on l_orderkey

3. Q6: Single-table scan of lineitem
   - Partition by l_shipdate ranges (highly selective)
   - Each thread computes partial sum independently
   - Final aggregation is trivial (single sum)

CRITICAL OPTIMIZATION STRATEGIES
---------------------------------

1. SORT ORDER SELECTION:
   - lineitem: Sort by l_shipdate (enables row group pruning for all 3 queries)
   - orders: Sort by o_orderdate (enables pruning for Q3)
   - customer: Sort by c_custkey (natural clustering for joins)

2. ROW GROUP SIZING:
   - lineitem: 50K rows/group (fine-grained pruning for date filters)
   - orders: 100K rows/group (balance between pruning and overhead)
   - Small tables: Use defaults (entire table fits in 1-2 groups)

3. DICTIONARY ENCODING:
   - Low-cardinality flags: l_returnflag, l_linestatus, o_orderstatus
   - Low-cardinality codes: c_mktsegment, l_shipmode, l_shipinstruct
   - High-cardinality fields: comments, names, addresses (disable dictionary)

4. COMPRESSION:
   - Large tables (lineitem, orders): snappy (fast decompression for scans)
   - Small tables: zstd (better ratio, decompress overhead negligible)

5. INDEX STRATEGY:
   - Build sorted indexes on all primary keys for fast lookups
   - Build sorted indexes on selective filter columns (l_shipdate, o_orderdate, c_mktsegment)
   - Build sorted indexes on foreign keys used in joins (l_orderkey, o_custkey)

EXPECTED PERFORMANCE IMPACT
----------------------------

Q1:
  - Row group pruning on l_shipdate: Skip ~5% of row groups
  - Column projection: Read 50% of columns
  - Expected speedup: 2-3x vs full scan

Q3:
  - Row group pruning: Skip ~70% of customer rows, ~70% of orders rows, ~25% of lineitem rows
  - Column projection: Read 25-44% of columns
  - Join optimization: Index lookups on c_custkey, o_orderkey
  - Expected speedup: 5-10x vs full scan + hash join

Q6:
  - Row group pruning on l_shipdate: Skip ~86% of row groups
  - Column projection: Read 25% of columns
  - Expected speedup: 10-15x vs full scan
