# Optimization Analysis

## Per-Query Assessment
- Q1 (11,955ms, FAIL): Floating-point precision error + suboptimal I/O and aggregation pattern

## Critical Fixes (must fix for correctness)
- [cpu_bound] Floating-point precision error → Implement Kahan summation
  - Current code (line 40): "no Kahan - double precision is sufficient" is WRONG
  - Validation shows 0.01 error in sum_disc_price (1058580922144.96 vs 1058580922144.97)
  - With 59M rows, rounding errors accumulate catastrophically
  - Fix: Replace naive summation in AggValues with compensated Kahan summation

## Recommended Optimizations

### 1. [cpu_bound] Kahan compensated summation
   Target: Q1 | Expected: Correctness fix (no perf change) | Risk: low
   Guidance:
   - Modify struct AggValues (lines 41-57) to add compensation terms:
     ```cpp
     struct AggValues {
         double sum_qty = 0.0, c_qty = 0.0;
         double sum_base_price = 0.0, c_base_price = 0.0;
         double sum_disc_price = 0.0, c_disc_price = 0.0;
         double sum_charge = 0.0, c_charge = 0.0;
         double sum_discount = 0.0, c_discount = 0.0;
         int64_t count = 0;
     ```
   - Replace all `agg.sum_X += value` with Kahan summation:
     ```cpp
     double y = value - agg.c_X;
     double t = agg.sum_X + y;
     agg.c_X = (t - agg.sum_X) - y;
     agg.sum_X = t;
     ```
   - Update merge() function (lines 49-56) to also use Kahan summation
   - Locations to fix: lines 226-231 (SIMD path), lines 253-258 (scalar path), lines 345-347 (merge)

### 2. [io_bound] Improve row group pruning selectivity
   Target: Q1 | Expected: 1.2x improvement | Risk: low
   Guidance:
   - Current pruning (lines 273-283) only checks `min <= threshold`, keeping 98.6% of row groups
   - Issue: Most row groups have at least one row with l_shipdate <= threshold, so min-based pruning is ineffective
   - Fix: Use BOTH min and max statistics to skip row groups where ALL rows fail filter:
     ```cpp
     // Skip row group if ALL rows are > threshold (i.e., min > threshold)
     if (s.has_min_max && s.min_int > date_threshold) {
         continue; // Skip this row group
     }
     relevant_rgs.push_back(s.row_group_index);
     ```
   - Expected: With ~5% late shipments uniformly distributed, this won't help much for Q1, but prepares for more selective queries

### 3. [cpu_bound] Reduce SIMD-to-scalar transition overhead
   Target: Q1 | Expected: 1.1-1.15x improvement | Risk: medium
   Guidance:
   - Current bottleneck (lines 216-232): SIMD computes disc_price and charge, but falls back to scalar for hash table updates
   - Problem: Store/load round-trip (_mm512_store_pd + array access) adds latency
   - Optimization: Use per-group SIMD accumulators before hash table update
     - Track returnflag/linestatus for last 8 rows
     - If all 8 rows belong to same group, accumulate in SIMD registers
     - Only update hash table after processing full chunk or group change
   - Risk: More complex code; benefit limited by low group cardinality (only 4 groups)
   - Alternative: Accept current design as good enough for low-cardinality aggregation

### 4. [cpu_bound] Thread-local aggregation with delayed merge
   Target: Q1 | Expected: 1.05x improvement | Risk: low
   Guidance:
   - Current implementation already uses thread-local hash tables (line 320, 344-347)
   - Current merge (lines 344-347) happens serially after all threads complete
   - Minor optimization: Pre-allocate final_map with expected capacity (4 groups) to avoid resizing
     ```cpp
     AggMap final_map;
     // Pre-allocate for 4 groups to avoid resize overhead
     for (int i = 0; i < 4; i++) final_map[i];
     ```
   - Benefit: Minimal, but eliminates resize during merge

### 5. [io_bound] Parquet I/O optimization
   Target: Q1 | Expected: 1.1x improvement | Risk: low
   Guidance:
   - Current code reads 59M rows from 1183 row groups (~50K rows/group)
   - Parquet reader.h uses parallel I/O (mentioned in header), but query processes sequentially after load
   - Potential improvement: Pipeline I/O and compute
     - Load row groups in batches (e.g., 100 at a time)
     - Start processing while next batch loads
   - Issue: Requires refactoring read_parquet_row_groups to support streaming
   - Risk: Moderate complexity increase; benefit depends on I/O vs compute balance

## Summary
**CRITICAL**: Fix Kahan summation immediately (correctness issue). Expected total speedup after all optimizations: 1.3-1.5x (from ~12s to ~8-9s). The query is fundamentally I/O bound (reading 59M rows) with limited optimization headroom given already-aggressive optimizations (AVX-512, multi-threading, open-addressing hash, row group pruning).
