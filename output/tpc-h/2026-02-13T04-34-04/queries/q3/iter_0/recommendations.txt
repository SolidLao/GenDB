## Recommended Optimizations

### 1. [io_bound] Row Group Pruning on orders.o_orderdate
   Target: Q3 | Expected: 2-3x improvement | Risk: low
   Guidance: Replace read_parquet (line 67-70) with:
   ```cpp
   auto orders_stats = get_row_group_stats(parquet_dir + "/orders.parquet", "o_orderdate");
   std::vector<int> orders_rg_indices;
   for (const auto& stat : orders_stats) {
       if (stat.has_min_max && stat.max_int < date_cutoff) {
           orders_rg_indices.push_back(stat.row_group_index);
       }
   }
   auto orders_table = read_parquet_row_groups(
       parquet_dir + "/orders.parquet",
       {"o_orderkey", "o_custkey", "o_orderdate", "o_shippriority"},
       orders_rg_indices
   );
   ```
   This skips row groups where max(o_orderdate) < date_cutoff, eliminating ~75% of I/O on orders table.

### 2. [io_bound] Row Group Pruning on lineitem.l_shipdate
   Target: Q3 | Expected: 1.5-2x improvement | Risk: low
   Guidance: Replace read_parquet (line 97-100) with:
   ```cpp
   auto lineitem_stats = get_row_group_stats(parquet_dir + "/lineitem.parquet", "l_shipdate");
   std::vector<int> lineitem_rg_indices;
   for (const auto& stat : lineitem_stats) {
       if (stat.has_min_max && stat.min_int > date_cutoff) {
           lineitem_rg_indices.push_back(stat.row_group_index);
       }
   }
   auto lineitem_table = read_parquet_row_groups(
       parquet_dir + "/lineitem.parquet",
       {"l_orderkey", "l_extendedprice", "l_discount", "l_shipdate"},
       lineitem_rg_indices
   );
   ```
   This skips row groups where min(l_shipdate) <= date_cutoff, eliminating ~25% of I/O on the largest table (60M rows).

### 3. [cpu_bound] Replace std::unordered_set with open-addressing hash set
   Target: Q3 | Expected: 1.2-1.3x improvement | Risk: medium
   Guidance: Replace std::unordered_set at line 47 with a custom open-addressing hash set or use a faster alternative (e.g., robin_hood::unordered_flat_set, absl::flat_hash_set). Current implementation uses std::unordered_set for building_customers (line 47) and lookup in the hot path (line 90). Open-addressing hash tables have 2-3x better cache locality and faster lookups than chaining-based std::unordered_set.

### 4. [cpu_bound] Replace std::unordered_map with open-addressing hash map
   Target: Q3 | Expected: 1.3-1.5x improvement | Risk: medium
   Guidance: Replace std::unordered_map at lines 84, 110, 162 with open-addressing alternatives. Three hash maps are used:
   - qualifying_orders (line 84): Build-side for join with lineitem
   - thread_maps (line 110): Thread-local aggregation maps
   - final_results (line 162): Global aggregation map
   All are accessed in hot paths. Open-addressing hash tables provide better cache efficiency and reduce memory overhead.

### 5. [query_structure] Fuse customer filter and hash set build
   Target: Q3 | Expected: 1.1x improvement | Risk: low
   Guidance: Lines 50-61 are already efficient - single-pass filter and build. No change needed. However, consider pre-sizing building_customers more accurately. Current reserve (customer_rows / 5) assumes 20% selectivity, which matches TPC-H specification for 5 market segments. This is correct.

### 6. [cpu_bound] Compiler optimizations - enable -O3 and native vectorization
   Target: Q3 | Expected: 1.2-1.4x improvement | Risk: low
   Guidance: Modify Makefile line 6 from `CXXFLAGS = -O2 -std=c++17 -Wall` to:
   ```makefile
   CXXFLAGS = -O3 -march=native -std=c++17 -Wall
   ```

   This enables aggressive optimizations and native SIMD instructions (AVX2/AVX-512). The revenue computation loop (line 129) and Kahan summation (lines 25-30) will benefit from auto-vectorization.

### 7. [cpu_bound] Manual SIMD vectorization of revenue computation

### 11. [cpu_bound] Optimize partial_sort limit

Verify this threshold is necessary - if revenues are always distinct enough, simplify to direct comparison for better branch prediction.

## Summary
Primary bottleneck is I/O-bound: implement row group pruning on orders.o_orderdate and lineitem.l_shipdate for 3-4x speedup. Secondary gains from replacing std::unordered containers with open-addressing alternatives (1.5-2x), compiler flags (-O3 -march=native, 1.2-1.4x), and optional SIMD vectorization (1.3-1.5x). Must fix missing CSV header first.

Combined expected speedup: 6-12x (from 7656ms to ~640-1275ms) with low/medium risk optimizations only.
