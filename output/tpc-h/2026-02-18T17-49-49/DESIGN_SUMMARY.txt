STORAGE DESIGN DECISIONS
─────────────────────────────────────────────────────────────────────────────

Format: Binary columnar (one .bin file per column)

Encoding Strategy:
  ✓ Integers:   int32_t (no compression needed)
  ✓ Decimals:   int64_t with scale_factor (100 for money, 1 for quantity)
  ✓ Dates:      int32_t days since epoch (1970-01-01)
  ✓ Strings:    Dictionary-encoded as int16_t codes
  ✓ Dictionary: Text files (code=value per line)

Sorting:
  - lineitem: Sorted by l_shipdate (improves zone-map pruning for Q1, Q3, Q6)
  - orders:   Sorted by o_orderdate (helps Q3 filter predicates)
  - Others:   Unsorted (access patterns are primarily hash joins)

Indexing:

  Zone Maps (Block-level min/max):
    - lineitem.l_shipdate:  600 zones, 16B each (9.6 KB)
    - orders.o_orderdate:   150 zones, 16B each (2.4 KB)
    ✓ Enables block-skipping for date range predicates
    ✓ Ultra-low overhead (<10 KB each)

  Hash Multi-Value (Join Keys):
    - lineitem.l_orderkey:  15M unique keys → 613 MB index
    - orders.o_custkey:     1.5M unique keys → 82 MB index
    - partsupp.ps_partkey:  2M unique keys → 79 MB index
    ✓ O(1) avg lookup, stores (key → row_offset, count)
    ✓ Load factor: 44-50% (efficient for memory access)

  Hash Single-Value (Primary Keys):
    - customer.c_custkey:   1.5M rows → 33 MB index
    - part.p_partkey:       2M rows → 33 MB index
    - supplier.s_suppkey:   100K rows → 2.1 MB index
    ✓ O(1) lookup for exact matches
    ✓ Fast dimension table lookups

QUERY OPTIMIZATION PATTERNS
─────────────────────────────────────────────────────────────────────────────

Q1 (Pricing Summary):
  - Full-table lineitem scan with filter on l_shipdate
  - Zone map skips ~1.5% of blocks (very selective)
  - Group by (l_returnflag, l_linestatus) → only 6 groups
  - Strategy: Vectorized filter + sorted aggregation

Q3 (Shipping Priority):
  - Customer → Orders → LineItem joins
  - Filters: c_mktsegment='BUILDING' (20% sel), o_orderdate < 1995-03-15 (48% sel)
  - Zone map on orders.o_orderdate, hash indexes on join keys
  - Strategy: Filter pushdown + hash multi-value lookups + top-10 radix sort

Q6 (Forecasting Revenue):
  - Single-table lineitem scan with 3 conjunctive filters
  - Combined selectivity: l_shipdate ∩ l_discount ∩ l_quantity = ~6%
  - Zone map on l_shipdate prunes blocks, then vectorized filter on qty/discount
  - Strategy: Block pruning + SIMD vectorized filtering

Q9 (Product Type Profit):
  - Complex 6-way join: part ⊲⊳ supplier ⊲⊳ lineitem ⊲⊳ partsupp ⊲⊳ orders ⊲⊳ nation
  - Pre-filter part on p_name LIKE '%green%' (5.4% selectivity)
  - Hash semi-joins with subquery elimination
  - Strategy: Early materialization of part subset, parallel multi-way joins

Q18 (Large Volume Customer):
  - Subquery: aggregate lineitem by l_orderkey for HAVING SUM(l_quantity) > 300
  - Main: customer ⊲⊳ orders ⊲⊳ lineitem with subquery filtering
  - Hash aggregation on subquery, hash semi-join on main
  - Strategy: Subquery materialization + hash joins + top-100 partial sort
