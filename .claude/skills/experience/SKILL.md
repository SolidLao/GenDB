# Experience: Correctness & Performance Rules
# Updated by DBA agent after each run. Checked by Code Inspector.

## Correctness (Critical — always check)

C1: DATE columns = int32_t epoch days. Use date_utils.h for all conversions. NEVER write custom date functions. [freq: 9, sev: HIGH]
C2: Dictionary-encoded columns — load _dict.txt at runtime. NEVER hardcode dictionary codes. [freq: 3, sev: HIGH]
C7: DATE + INTERVAL — use gendb::add_years/add_months/add_days. NEVER +-N*365 or +-N*30. [freq: 4, sev: HIGH]
C9: Hash table capacity MUST use next_power_of_2(count * 2) for ≤50% load factor. Join build-side hash tables: size for full BUILD-SIDE cardinality per thread. Aggregation thread-local hash tables: size for full GROUP cardinality (use FILTERED count per P23, not raw table size). **Memory guard**: always compute total_agg_mem = nthreads × cap × slot_size and compare against hardware LLC — see aggregation-optimization skill: Memory Budget Gate for strategy selection. [freq: 9, sev: HIGH]
C11: Call gendb::init_date_tables() once at top of main() before any extract_year/month/day. [freq: 5, sev: HIGH]
C15: Hash map key MUST include ALL GROUP BY columns. Missing dimension → wrong aggregation groups. Diagnostic: if value AND count are both off by the SAME exact integer factor N simultaneously (e.g. both exactly 2× or 0.5×), a GROUP BY key dimension is missing. [freq: 6, sev: HIGH]
C19: Zone-map block skip ONLY on columns with zone-map indexes per Query Guide. Never on unsorted columns. [freq: 2, sev: HIGH]
C20: NEVER memset() for multi-byte sentinels. memset(buf, 0x80, n) → 0x80808080 ≠ INT32_MIN. Use std::fill(). [freq: 6, sev: HIGH]
C24: ALL hash tables MUST use bounded probing: for-loop with probe < cap, not unbounded while. [freq: 10, sev: HIGH]

## Correctness — Data-Dependent

C13: Optimizer MUST NOT change date constants from passing iterations. Preserve original values. [freq: 2, sev: HIGH]
C18: Dictionary output — decode code to string via dict[code].c_str(), not raw int16_t. [freq: 4, sev: MED]
C22: Database path — use EXACT path from Query Guide. Never derive from scale factor string. [freq: 2, sev: MED]
C23: Repeated crash/timeout with no [TIMING] → suspect C9 or C20. Audit ALL hash tables. [freq: 4, sev: HIGH]

C29: Large-magnitude double columns (values up to 10^15-10^16) — double has only ~15.95 sig-digit precision. Accumulating and printing %.2f produces last-digit mismatches vs reference (+-0.01). MANDATORY for SUM(single_column) where column max >= 10^10. Fix: accumulate as int64_t cents (iv = llround(v * 100.0); sum += iv) then output as sum/100 + "." + abs(sum%100). CRITICAL: if storage_design.json annotates a column with "C29 WARNING: SUM must accumulate as int64_t cents", treat as HARD CONSTRAINT. **SCOPE LIMIT**: C29 applies ONLY to SUM(single_column) — NOT to derived expressions like SUM(ep*(1-disc)). For multi-column derived expressions, use long double (see C35). Seen on SEC EDGAR num.value column (max 1e16). [freq: 4, sev: HIGH]

C35: int64_t cents (C29) MUST NOT be applied to derived multi-column expressions. C29 is valid ONLY for SUM(single_column). For SUM(expr) where expr = col_a * (1 ± col_b) or similar multi-column product, use `long double` accumulation instead. Root cause: `llround(ep * 100.0 * (1.0 - disc))` — each FP multiply loses bits before conversion; 60M rows × ~0.03 per-row FP error = ~$1800 accumulated error on sum_disc_price. Diagnostic: validation fails on sum_disc_price or sum_charge with abs_diff $100–$2000; same code works correctly for sum_base_price (single column). Fix: declare AggSlot monetary fields as `long double` for derived expressions; accumulate `ep * (1.0 - disc)` directly without ×100 intermediate. TPC-H Q1 iter_2 failed for this reason. [freq: 1, sev: HIGH]

## Correctness — Conditional (int64_t DECIMAL encoding only)

C3: DECIMAL scale_factor — every threshold, AVG divisor, HAVING comparison must use scale_factor from Query Guide. [freq: 2, sev: HIGH]
C5: SUM(col_a * col_b) with int64_t — divide by scale after each multiplication. [freq: 2, sev: HIGH]
C14: Revenue formula: ep * (scale_factor - discount_column) / scale_factor. Keep canonical pattern. (TPC-H example — the underlying PRINCIPLE of canonical formula preservation is general.) [freq: 2, sev: MED]

C30: Exact N× factor mismatch across ALL aggregate columns simultaneously = missing GROUP BY key. Diagnostic signature: expected_value / actual_value = exact integer N AND expected_count / actual_count = same N. This means N distinct groups with the same partial key were merged into one (summing N copies of the data). Fix: audit hash map key struct against full GROUP BY clause; add missing dimension. [freq: 2, sev: HIGH]

## Correctness — Optimizer Behavioral

C25: rows_actual == 0 with rows_expected > 0 → filter elimination. Revert to last passing filter logic. Without a passing baseline, the optimizer has no safe anchor — establishing a correct (even slow) baseline is first priority. [freq: 3, sev: HIGH]
C26: Correct row count but wrong values after join restructure → revert build_joins + main_scan to last passing iteration. [freq: 2, sev: HIGH]
C27: Pre-built index refactoring requires ALL 5 elements updated atomically: (1) struct definition (PSSlot etc.), (2) mmap call for index file (xxx_index_raw), (3) header parse (cap, mask=cap-1, ht pointer from raw+4), (4) hash function name, (5) ALL probe loop sites. Missing any one → compile fail. Error signatures: 'XXXSlot does not name a type', 'xxx_raw not declared in this scope', 'is_mask/is_cap/is_ht not declared in this scope'. Most common missing piece: element (3) — file is mmap'd but cap/mask/ht never extracted from header before probe loop. [freq: 2, sev: HIGH]
C28: Hash function rename must propagate to ALL call sites. Error: 'ps_hash not declared; did you mean cps_hash?' — renamed in definition but probe loop still calls old name. [freq: 1, sev: HIGH]

C31: CSV output — always double-quote ALL string/dictionary output columns. Unquoted strings containing commas cause CSV column shift: parser splits at the comma, shifting every subsequent column by +1. Diagnostic: validation shows string_mismatch on multiple adjacent columns with correct row count; expected value in col N appears in col N+1 actual. Fix: printf("\"%s\"", str.c_str()) for EVERY string column, regardless of expected content. [freq: 3, sev: HIGH]

C32: Pre-built index header parse must declare ALL THREE variables at function scope before any loop that uses them: (1) uint32_t X_cap = *(const uint32_t*)X_raw; (2) uint32_t X_mask = X_cap - 1; (3) const XSlot* X_ht = (const XSlot*)(X_raw + 4); If declared inside a data_loading lambda, if-block, or #pragma section that is not visible to main_scan, the compiler reports 'X_mask not declared in this scope'. Fix: hoist all three declarations to run_query() function scope. [freq: 2, sev: HIGH]

C33: TOP-N queries — always include a stable unique tiebreaker as the LAST sort key. When ORDER BY primary key has ties, std::sort / std::partial_sort produces non-deterministic ordering that differs from reference. Fix: append unique secondary key (cik, entity_id, adsh, or lexicographic name) to every TOP-N comparator: if (a->val != b->val) return a->val > b->val; return a->id < b->id; [freq: 4, sev: HIGH]

C34: Optimizer MUST lock in thread-local aggregation once validated — NEVER revert to shared map. Once thread-local aggregation_merge achieves >20% speedup over a shared-map baseline, preserve the thread-local struct, per-thread map allocation, and merge loop in ALL subsequent optimization iterations. Only hash function, key layout, or capacity may be tuned. Diagnostic for reversion: aggregation_merge spikes by >50% vs best previously seen value. Fix: track best aggregation_merge timing; if current iter's aggregation_merge > 1.5× best_seen, discard and revert. [freq: 2, sev: HIGH]

C36: Partitioned hash aggregation — p1_part() and p1_hash() MUST use ORTHOGONAL bit ranges of the same hash function. If both use low bits, keys in partition p share the same low-K bits → all cluster in P1_CAP/NUM_PARTITIONS slots of the merge map → catastrophic probe chains → timeout (300s). Fix: p1_part MUST use HIGH bits: `(key * KNUTH) >> (32 - log2(NUM_PARTITIONS))`; p1_hash MUST use LOW bits: `(key * KNUTH) & P1_MASK`. Diagnostic: query with NUM_PARTITIONS partitioned aggregation times out with no [TIMING] output and no compile error. TPC-H Q18 iter_0: 64 partitions, both functions used low bits → 28× merge map overload → 300s timeout. Confirmed working: 2026-02-24 run Q18 iter_4 achieved 115ms subquery_precompute with correct orthogonal bits. [freq: 2, sev: HIGH]

## Performance (Suggestions — non-blocking)

P1: Replace std::unordered_map with open-addressing for >1000 entries. 2-5x faster. [impact: 2-5x, freq: 6]
P2: Don't copy mmap'd data into vectors. Use reinterpret_cast<T*> for zero-copy. [impact: 500ms+, freq: 4]
P11: Use pre-built hash indexes via mmap. Zero build cost vs 100-500ms runtime build. [impact: 100-500ms, freq: 8]
P12: Track best (timing_ms, source). If >20% regression, revert and try different direction. Q18 2026-02-24: iter_1 +92% regression (416ms vs 217ms); revert to iter_2 (226ms) then breakthrough at iter_4 (136ms). [impact: N/A, freq: 7]
P13: Zone-map-guided selective madvise when <50% blocks qualify. Don't WILLNEED entire column. [impact: 2-5x I/O, freq: 3]
P16: data_loading >40% hot runtime OR >150ms absolute → apply zone-map-guided prefetch AND concurrent multi-index madvise (P27) AND parallel dict loading. Q6 current run: data_loading=258ms — absolute threshold triggers. SEC-EDGAR run: Q6 iter_5 combined P27+parallel dict → 132ms→31ms (-76%). TPC-H Q9: data_loading=180ms → 0.45ms after zone-map selective madvise applied. [impact: 30-76%, freq: 6]
P17: aggregation_merge >20ms → switch to thread-local per-key reduction then single merge pass. Universal win: Q1 (49ms→2ms), Q6 (320ms→3ms), Q24 (167ms→1ms). Apply by default in code generation. WARNING: shared unordered_map merge across 64 threads = catastrophic (Q6 iter_1: 46,800ms). [impact: 10-46800ms, freq: 12]
P20: Thread-local aggregation_merge should be the DEFAULT pattern — a single shared aggregation map causes severe contention at scale. Thread-local reduction eliminates 10-300ms+ overhead with no correctness risk. Catastrophic threshold is LOWER than expected: 50K groups × 64 threads → 350ms. See aggregation-optimization skill for merge strategy selection. [impact: 10-46800ms, freq: 6]
P21: Anti-join or semi-join on build side >1M rows → use bloom filter pre-filtering. Bloom uses ~1.5 bytes/item vs 8-16 bytes/item for hash table. Eliminates 80-99% of probes. [freq: 1, sev: HIGH]
P23: Hash table capacity MUST be based on FILTERED row count, not raw table size. Over-sizing wastes L3 cache and causes LLC thrashing. Example: 548K filtered rows → cap = next_power_of_2(548K * 2) = 1M (16MB), not 9.6M * 2 = 33M (512MB). [freq: 2, sev: MED]
P25: Thread-local occupied-slot tracking + parallel sort-merge can still be slow at high thread counts if per-thread maps are oversized. Fix: (1) apply P23 to aggregation (size AGG_CAP to actual group cardinality); (2) use merge strategy from aggregation-optimization skill (partitioned merge or scan-time partitioned aggregation based on total memory vs LLC). [impact: 100-400ms, freq: 2]
P27: Large multi-index data_loading (2+ pre-built mmap index files each >30MB) → use concurrent madvise across index files. Sequential madvise(MADV_WILLNEED) serializes I/O prefetch; parallel sections let the OS prefetch all indexes simultaneously. Trigger: data_loading >150ms absolute with 2+ index files. Implementation: #pragma omp parallel sections { section: madvise(idx1_raw, sz1, MADV_WILLNEED); section: madvise(idx2_raw, sz2, MADV_WILLNEED); }. [impact: 30-76% data_loading, freq: 3]
P28: Thread-local aggregation total memory (nthreads × AGG_CAP × slot_size) exceeding LLC → initialization dominates (page faults + std::fill) and merge scans thrash cache. Fix: evaluate aggregation-optimization skill: Memory Budget Gate — scan-time partitioned aggregation eliminates large per-thread maps entirely. [impact: 100-400ms, freq: 2]
P30: Compact pre-filter HT build for selective dimension predicates. When a dimension table has a WHERE predicate with selectivity <50%, split the build phase: (1) compact_ht_build — scan dimension, apply all WHERE filters, collect qualifying PKs into a flat compact array; (2) build_joins — build hash table ONLY over the compact filtered set. Trigger: build_joins >200ms AND dimension filter estimated to pass <50% of rows. TPC-H Q3: build_joins 334ms→46ms. [impact: 2-7x build_joins, freq: 1]
P31: Plateau detection — terminate optimization after 3 consecutive non-improving iterations. Track (best_timing_ms, consecutive_no_improve_count). After each iteration: if current_timing ≤ best_timing * 1.05, increment counter; else reset. If counter ≥ 3, emit PLATEAU_STOP and return best iteration's source. Also log the dominant bottleneck phase at plateau to guide future runs. TPC-H Q18 prior run: iters 3–5 all within 3% (1035–1068ms) — 3 wasted iterations. TPC-H Q3 2026-02-24: iters 2–4 all within 1% (137–138ms) — wasted 2 iterations before breakthrough at iter_5. NOTE: plateau may precede a structural breakthrough — on PLATEAU_STOP, also attempt one pivot to a categorically different technique before giving up (see P39). [impact: saves 2-4 wasted iterations per plateaued query, freq: 3]
P32: Dense-key bitmap for semi-join — when a dimension PK is a dense sequential integer range [1, N] and N/8 bytes fits L2 cache (N < ~2M), use a flat bitmap for O(1) semi-join lookup. Replaces ~5 cache misses per hash probe with a single L2-resident bit test. TPC-H Q3 + Q9: 187KB/250KB bitset replaced multi-MB hash index. [impact: eliminates hash probe overhead, freq: 1]
P33: Probe-aggregate fusion — when GROUP BY key ⊇ join key, fuse join probe and aggregation: accumulate aggregate values directly into join HT slots during probe scan via atomic CAS double-add. Eliminates intermediate tuple materialization and aggregation_merge phase. TPC-H Q3: 79ms (fused) vs 188ms (partition-buffer merge). [impact: eliminates aggregation_merge + reduces main_scan 30-50%, freq: 1]
P34: SoA layout + CAS-based concurrent HT — for hash tables serving both join build and probe-aggregate fusion: struct-of-arrays layout (separate atomic keys[], metadata[], atomic accumulators[]) with CAS-based parallel insert during build. Benefits: parallel build + atomic-only probe accumulation. See hash-tables skill: CAS-Based Concurrent Insert. [impact: enables parallel build + probe-aggregate fusion, freq: 1]

P35: LIKE '%keyword%' on large dictionary file — use mmap raw scan + memmem(), NOT vector<string>+find(). Avoids 2M+ heap string allocations and per-entry cache-unfriendly access. Algorithm: (1) mmap _dict.txt as raw bytes, (2) global memmem() scan to collect byte offsets of keyword, (3) two-pointer newline-count to map offsets to line numbers (code values), (4) bitmap-set qualifying part PKs. Trigger: dim_filter >100ms with LIKE '%keyword%' on dict-encoded column with >100K dict entries. TPC-H Q9 prior run: 313ms → 57ms (-82%). TPC-H Q9 2026-02-24: 102ms → 13.7ms (-87%). [impact: 2-7x dim_filter, freq: 2]

P36: Static mmap caching for hot runs — use `static const T* s_col = nullptr;` at file scope, guarded by `if (!s_col) { mmap... }`. Eliminates re-mmap and re-madvise overhead on repeated hot timing invocations. Trigger: data_loading >5ms on hot runs when columns are already OS-cached from prior cold run. TPC-H Q6: data_loading 18ms → 0.1ms between iter_0 and iter_1. TPC-H Q3 2026-02-24: data_loading 63ms → 0.1ms (iter_0 → iter_1, -99.8%). Safe: gendb_dir is constant across calls in all GenDB benchmarks. [impact: 5-63ms data_loading on hot runs, freq: 2]

P37: Partitioned hash aggregation — P1_CAP must be sized to per-partition unique key count, NOT total unique keys. Over-sizing causes page-fault stall during merge map initialization. Formula: P1_CAP = next_power_of_2(total_unique_keys / NUM_PARTITIONS * 2). Example: TPC-H lineitem 15M unique orderkeys / 64 partitions * 2 = 468750 → P1_CAP = 524288 (6.3MB/partition). Using over-sized P1_CAP=524K with wrong base caused Q18 subquery_precompute ~1000ms in prior run. Correctly sized in 2026-02-24 run → 115ms. Diagnostic: subquery_precompute >500ms with partitioned aggregation; P1_CAP × NUM_PARTITIONS × slot_size >> L3 cache. [impact: 5-20× subquery_precompute, freq: 2]

P38: Shared XADD aggregation for integer accumulators — single shared hash table with __atomic_fetch_add beats partitioned approaches at high cardinality (>100K groups). See aggregation-optimization skill: Shared XADD Aggregation. [impact: 5-17x, freq: 1]

P39: Plateau-pivot — after 3-iter plateau (P31), force a STRUCTURAL technique change, not incremental tuning. Incremental tuning of the same code structure (adjusting HT sizes, loop ordering, prefetch distances) exhausts quickly and all land within 1% of each other. A breakthrough requires a categorically different approach: change aggregation strategy (thread-local → partitioned → XADD), fuse scan phases, switch join algorithm, or add a new dim_filter stage. TPC-H Q3 2026-02-24: iters 2–4 stagnant at 137ms; iter_5 switched technique (new ht_init phase, dim_filter halved 4.7→1.0ms, build_joins −7ms) → 99ms breakthrough. Emit 'PLATEAU_PIVOT' log; select highest-impact untried technique from experience skill before giving up. [impact: unlocks 20-40% further improvement after plateau, freq: 1]

P40: main_scan SIMD — when main_scan >30ms and is >80% of total after 2+ optimization rounds, consider AVX2 vectorization. Q1 (44.7ms), Q3 (47.5ms), Q6 (30.1ms), Q9 (33.8ms) all plateau with main_scan dominant. Arithmetic intensity is low (FMA of double columns) making auto-vec feasible. Prerequisite: SOA layout (separate column arrays, no struct-of-rows), counted inner loop, no pointer aliasing. Check: -O3 -march=native already in compile flags. If auto-vec fails, use _mm256_loadu_pd + _mm256_fmadd_pd for 4-wide double SIMD. Trigger only after HT, join, and agg phases are already optimized. [impact: 2-4x theoretical main_scan, freq: 1]
